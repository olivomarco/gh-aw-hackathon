## Goal

Watch every push to `main` and alert the team if the changed files contain
patterns that commonly indicate breaking changes: removed exports, renamed
functions, deleted API endpoints, or schema field removals.

This is the canonical **push-event + conditional create-issue** pattern.
The key teaching moment: use `noop` when the push is clean — do not spam
the issue tracker on every green commit.

## Steps

1. **Collect the diff** using bash tools:
   ```
   git diff HEAD~1 HEAD --stat
   git diff HEAD~1 HEAD -- src/ api/ *.go *.ts *.py
   ```
   Store the output mentally as the "change set".

2. **Analyse the change set** for these risk signals:

   | Signal | Examples |
   |--------|---------|
   | Removed function/method signature | `-func`, `-def`, `-export function` |
   | Renamed public symbol | old name removed AND new name added in same file |
   | Deleted API route | `-app.get(`, `-router.delete(`, `-@Delete(` |
   | Schema field removal | `-  field:`, `-  "property":` from JSON/YAML schema files |
   | `BREAKING CHANGE:` in commit message | literal substring match |

3. **If no risk signals are found**, call `noop` with reason
   `"No concerning changes detected in this push."` and stop.

4. **If one or more risk signals are found**, create an alert issue
   using the `create-issue` safe-output with the template below.

## Output format

**Issue title:** `[watcher] Possible breaking change — {commit SHA, first 7 chars}`

**Issue body:**

```markdown
## 🔭 Watcher Alert — Possible Breaking Change

**Commit:** {SHA} by @{author}
**Branch:** main
**Pushed at:** {timestamp}

### Risk signals detected

{For each signal found, one bullet:}
- **{Signal type}**: `{file path}` — {one-sentence description of what changed}

### Changed files summary

```
{git diff --stat output, max 20 lines}
```

### Recommended actions

- [ ] Review the diff for unintended API surface changes
- [ ] If intentional: update the CHANGELOG and bump the major version
- [ ] If accidental: revert the commit or add a compatibility shim
- [ ] Close this issue once the change is reviewed

_Auto-generated by The Watcher on push to main._
```

## Constraints

- Analyse **only** the diff between `HEAD~1` and `HEAD` (the last single commit).
  Do not look further back — that is out of scope for a push trigger.
- Do **not** flag changes to `test/`, `*_test.go`, `*.spec.ts`, or `*.test.py` —
  test files are not part of the public API surface.
- Create at most **one** issue per push. If multiple signals are found, group them
  in a single issue rather than creating one per signal.
- The issue body must stay under 4 000 characters.
