---
on:
  # Manual trigger only — this is a learning exercise, not a production workflow.
  workflow_dispatch:
    inputs:
      dry_run:
        description: "Set to 'true' to produce a noop output and skip real writes"
        required: false
        default: "false"

permissions:
  # Minimal read permissions. safe-outputs: create-issue owns the write path.
  contents: read
  issues: read

safe-outputs:
  create-issue:
    # Only one demo issue at a time; close the previous one automatically.
    max: 1
    close-older-issues: true
    labels: [safe-and-sound, automated]
    title-prefix: "[safe-and-sound] "
  noop:
    # Called when dry_run=true OR when there is genuinely nothing to write.
    reason: "Dry-run mode — no writes performed."

engine: copilot
---

## Goal

Demonstrate the two fundamental safety mechanisms in gh-aw:

1. **Permission scoping** — the workflow frontmatter grants only the minimum GitHub
   permissions needed; safe-outputs handles all write operations under a separate,
   bounded gate.
2. **The `noop` exit protocol** — every gh-aw workflow *must* call a safe-output
   before exiting. `noop` is the safe way to say "I ran successfully but had nothing
   to write."

This workflow teaches both concepts by running a deliberate "health check" and
demonstrating both paths.

## Steps

1. **Read the current repository state** using the GitHub MCP tools:
   - Count of open issues
   - Count of open pull requests
   - Repository visibility (public or private)

2. **Check the `dry_run` input.** If the workflow was triggered with
   `dry_run: true`, skip step 3 and proceed directly to step 4 (the noop path).

3. **Create a health check issue** (write path) using the `create-issue`
   safe-output. Format the issue using the template in the Output format section.

4. **If dry_run is true**, call `noop` with the reason:
   `"Dry-run mode — no writes performed."` and explain in a brief agent note that
   the data was collected but no issue was created.

## Output format

**Issue title:** `[safe-and-sound] Repository Health Check — {YYYY-MM-DD}`

**Issue body:**

```markdown
## 🛡️ Safe & Sound — Repository Health Check

This issue was created by the **Safe & Sound** workflow to demonstrate
gh-aw's `safe-outputs` gate.

### What just happened

| Mechanism | Value |
|-----------|-------|
| Workflow permissions | `contents: read`, `issues: read` |
| Write method | `safe-outputs: create-issue` (bounded gate) |
| Engine | Copilot |

### Repository snapshot at time of run

| Metric | Count |
|--------|-------|
| Open issues | {N} |
| Open pull requests | {N} |
| Visibility | {public / private} |

### Key takeaways

- The agent **cannot** write to GitHub directly — it can only call safe-outputs.
- `permissions: contents: read` means the underlying GitHub token is read-only.
- This issue was created through the `create-issue` safe-output, not raw API calls.
- If `dry_run=true` had been set, `noop` would have been called and this issue
  would not exist.

_Auto-generated by the Safe & Sound challenge workflow._
```

## Constraints

- Do **not** add any permissions beyond `contents: read` and `issues: read` to
  the frontmatter — this workflow intentionally demonstrates minimal scoping.
- Do **not** attempt to call the GitHub REST API directly. Use safe-outputs only.
- The noop path must always be reachable via the `dry_run` input — this is the
  teaching moment, not a fallback.
